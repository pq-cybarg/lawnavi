Post-quantum crypto & an encrypted matter vault — entirely on-device.

Law Navi runs fully in your browser. This optional companion process adds validated-grade, post-quantum cryptography and a confidential matter vault for lawyers and clients who need it — without anything ever leaving the local machine. No cloud, no telemetry, no account.

Why it exists

Attorney work product and client communications are confidential and often privileged. A browser tab is fine for studying the law, but for storing real matter data you want strong encryption at rest, authenticated integrity, post-quantum signatures, and an auditable trail — all under your sole control. The sidecar provides exactly that, as a single dependency-light Node process bound to 127.0.0.1 only.

Privacy model: the sidecar listens on localhost only, requires a bearer token (printed once at startup), stores everything encrypted at rest in a directory you choose, keeps the master key only in memory (derived from your passphrase), and never makes an outbound connection except — if you explicitly configure one — to a local LLM you run yourself.

PQC is preferred everywhere it is available; classical algorithms are used only where compatibility requires it (for example, the in-browser vault uses AES-GCM + PBKDF2 because browsers do not yet expose argon2id / ML-DSA in WebCrypto).

Quick start

# 1) get the two files: lawnavi-local.mjs and package.json
# 2) install the audited pure-JS crypto deps (@noble/hashes, @noble/post-quantum)
npm install

# 3) run it (a passphrase unlocks the encrypted matter vault)
LAWNAVI_PASSPHRASE='a long passphrase you remember' node lawnavi-local.mjs

# optional: force OpenSSL FIPS mode for the AES/SHA paths
LAWNAVI_PASSPHRASE='…' LAWNAVI_FIPS=1 node lawnavi-local.mjs

Then, in the main app, open 🔒 → “Detect local FIPS/PQC sidecar”. The startup banner prints a one-time bearer token — paste it into your own client/scripts to use the protected endpoints.

On FIPS

The sidecar can run under OpenSSL 3's FIPS provider (LAWNAVI_FIPS=1) so the AES-GCM and SHA operations use validated paths. Note that argon2id — chosen for its memory-hardness against modern password attacks — is deliberately not a FIPS-approved KDF, so a strict-FIPS-only deployment would substitute PBKDF2. Browser WebCrypto is never a FIPS-validated module; for validated FIPS, run the operations here under a FIPS-built Node/OpenSSL. This is an honest, security-first set of trade-offs, not a compliance claim.
